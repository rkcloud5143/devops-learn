# Observability — Metrics, Logs & Traces Explained 👁️

The three pillars of observability, explained simply.

---

## The Pizza Shop Analogy

```
Your pizza shop is busy. Something's wrong — orders are slow.
How do you figure out what's happening?

📊 METRICS  = The dashboard on the wall
   "Orders per hour: 50"  "Average wait time: 25 min"  "Oven temp: 450°F"
   Numbers over time. Tells you WHAT is happening.

📋 LOGS     = The notebook where cooks write notes
   "3:15pm — Oven 2 stopped heating"
   "3:16pm — Error: cheese delivery didn't arrive"
   Text records. Tells you WHY something happened.

🔗 TRACES   = Following one order from start to finish
   "Order #42: Received (0s) → Kitchen (2s) → Oven (8min) → Boxing (30s) → Delivered (15min)"
   Shows the full journey. Tells you WHERE the slowdown is.
```

---

## Metrics 📊

**What:** Numbers measured over time.

**Think of it as:** A car dashboard — speed, fuel, RPM, temperature.

```
Examples:
  CPU usage:           72%
  Memory usage:        4.2 GB / 8 GB
  Requests per second: 500
  Error rate:          0.5%
  Response time (p95): 200ms
  Disk usage:          65%
  Active connections:  1,200
```

**Key characteristics:**
- Numeric values
- Collected at regular intervals (every 15s, 30s, 1m)
- Cheap to store (just numbers)
- Great for dashboards and alerts
- Answers: "Is something wrong RIGHT NOW?"

**Tools:** Prometheus, CloudWatch, Datadog, Grafana

**Example alert:**
```
IF cpu_usage > 90% FOR 5 minutes → Page on-call engineer
IF error_rate > 5% FOR 2 minutes → Send Slack alert
```

## Logs 📋

**What:** Timestamped text records of events.

**Think of it as:** A diary — records what happened and when.

```
Examples:
  2024-01-15 10:23:45 INFO  User john@example.com logged in
  2024-01-15 10:23:46 INFO  GET /api/orders 200 45ms
  2024-01-15 10:23:47 ERROR Connection to database refused: timeout after 5s
  2024-01-15 10:23:47 WARN  Retrying database connection (attempt 2/3)
  2024-01-15 10:23:48 ERROR Failed to process order #1234: database unavailable
```

**Key characteristics:**
- Text (structured or unstructured)
- Generated by application code
- Detailed context (error messages, stack traces)
- Expensive to store at scale (lots of text)
- Answers: "WHY did this error happen?"

**Log levels:**
```
DEBUG → Detailed developer info (noisy, dev only)
INFO  → Normal operations ("User logged in", "Order created")
WARN  → Something unexpected but not broken ("Retry attempt", "Slow query")
ERROR → Something failed ("Database connection refused", "Payment failed")
FATAL → App is crashing ("Out of memory", "Cannot bind to port")
```

**Tools:** Elasticsearch + Kibana (ELK), Loki + Grafana, CloudWatch Logs, Datadog Logs

## Traces 🔗

**What:** The journey of a single request across multiple services.

**Think of it as:** A GPS tracker on a package — shows every stop and how long it spent there.

```
Without traces (microservices):
  User says: "The app is slow"
  You have 15 services. Which one is slow? 🤷

With traces:
  Request #abc123:
  ┌─ API Gateway (2ms)
  │  └─ Auth Service (15ms)
  │     └─ User Service (8ms)
  │        └─ Database query (3ms)
  │  └─ Order Service (1,200ms) ← 🐌 HERE'S THE PROBLEM
  │     └─ Payment API (1,150ms) ← External API is slow!
  │     └─ Database query (5ms)
  │  └─ Notification Service (20ms)
  └─ Total: 1,250ms
```

**Key concepts:**
```
Trace    = The full journey of one request (end-to-end)
Span     = One step in the journey (one service, one DB call)
Trace ID = Unique ID that links all spans together

Every service passes the Trace ID to the next service.
This is how you connect the dots across 15 microservices.
```

**Key characteristics:**
- Shows request flow across services
- Identifies bottlenecks (which service is slow?)
- Shows dependencies (Service A calls B calls C)
- Answers: "WHERE is the slowdown?"

**Tools:** Jaeger, AWS X-Ray, Datadog APM, Zipkin, Tempo

## How They Work Together

```
Alert fires: "Error rate > 5%"                    ← METRIC told you WHAT

You check logs: "Connection refused to payment-api" ← LOG told you WHY

You check traces: Request spent 5s waiting for       ← TRACE told you WHERE
                  payment-api which timed out

Root cause: Payment API is down.
```

### Real-World Example

```
1. 📊 METRIC: CloudWatch alarm fires
   "Error rate jumped from 0.1% to 8% in the last 5 minutes"

2. 🔗 TRACE: You open Jaeger/X-Ray, look at failed requests
   "All failing requests are timing out at the order-service → database span"
   "Database calls taking 30 seconds instead of 50ms"

3. 📋 LOG: You search logs for order-service
   "ERROR: too many connections to PostgreSQL (max: 100, current: 100)"
   "WARN: connection pool exhausted, waiting for available connection"

4. 🔧 FIX: Database connection pool is maxed out
   → Increase max connections
   → Add connection pooling (PgBouncer / RDS Proxy)
   → Scale read replicas
```

---

## Correlation — Connecting the Three

```
The magic: Trace ID appears in ALL three.

Metric:  error_rate{service="order-service"} = 8%
           → Click to see traces for this service

Trace:   trace_id=abc123 → order-service → db-query (30s timeout)
           → Click to see logs for this trace

Log:     [trace_id=abc123] ERROR: connection pool exhausted
           → Full context of what happened

This is why you need ALL THREE, not just one.
```

## Quick Comparison

```
             Metrics          Logs              Traces
─────────    ──────────       ──────────        ──────────
What         Numbers          Text              Request flow
Answers      WHAT happened    WHY it happened   WHERE it happened
Format       Numeric          Text/JSON         Spans + timing
Storage      Cheap            Expensive         Medium
Cardinality  Low              High              Medium
Best for     Alerts,          Debugging,        Bottleneck
             dashboards       root cause        finding
Tools        Prometheus       ELK, Loki         Jaeger, X-Ray
             CloudWatch       CloudWatch Logs   Datadog APM
             Datadog          Datadog Logs      Tempo
```

---

*You need all three. Metrics tell you something's wrong. Traces tell you where. Logs tell you why. 👁️*
