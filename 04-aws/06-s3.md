# AWS S3 — Deep Dive

## CORE CONCEPTS

```
┌─── S3 (Global Service) ──────────────────────────────────────┐
│                                                               │
│  Bucket = container for objects (globally unique name)       │
│  Object = file + metadata (up to 5TB per object)            │
│  Key    = full path of the object (s3://bucket/folder/file) │
│                                                               │
│  There are no real "folders" — it's a flat namespace         │
│  s3://my-bucket/images/photo.jpg                             │
│  Key = "images/photo.jpg"                                    │
│                                                               │
│  Bucket naming rules:                                        │
│  - 3-63 characters, lowercase, numbers, hyphens             │
│  - Must be globally unique across ALL AWS accounts           │
│  - Must start with letter or number                          │
│  - No underscores, no uppercase                              │
└───────────────────────────────────────────────────────────────┘
```

## VERSIONING

```
Bucket with versioning ENABLED:

PUT photo.jpg (v1) → stored as version 1
PUT photo.jpg (v2) → stored as version 2 (v1 still exists)
DELETE photo.jpg   → adds "delete marker" (v1 and v2 still exist!)

┌─────────────────────────────────────────────┐
│  photo.jpg                                   │
│  ├── Version 3 (current)                    │
│  ├── Version 2                              │
│  └── Version 1                              │
│                                              │
│  To truly delete: delete specific version ID │
│  To restore: delete the "delete marker"      │
└─────────────────────────────────────────────┘

Use for: accidental deletion protection, audit trail
Warning: increases storage costs (all versions stored)
```

## ENCRYPTION

```
┌─── Server-Side Encryption (data at rest) ────────────────────┐
│                                                               │
│  SSE-S3 (default):                                           │
│  - AWS manages keys entirely                                 │
│  - AES-256 encryption                                        │
│  - No extra config needed                                    │
│  - Header: "x-amz-server-side-encryption": "AES256"         │
│                                                               │
│  SSE-KMS:                                                    │
│  - You control keys via AWS KMS                              │
│  - Audit trail (who used which key when)                     │
│  - Can use customer managed key (CMK)                        │
│  - Header: "x-amz-server-side-encryption": "aws:kms"        │
│  - Limitation: KMS API rate limits                           │
│                                                               │
│  SSE-C:                                                      │
│  - YOU provide the encryption key                            │
│  - AWS does NOT store your key                               │
│  - Must use HTTPS                                            │
│  - You manage key rotation                                   │
│                                                               │
│  Client-Side Encryption:                                     │
│  - YOU encrypt before uploading                              │
│  - YOU decrypt after downloading                             │
│  - AWS never sees unencrypted data                           │
└───────────────────────────────────────────────────────────────┘
```

## LIFECYCLE RULES

```
Automatically transition or delete objects based on age

Example:
┌──────────────────────────────────────────────────────────────┐
│                                                               │
│  Day 0:   Object created in S3 Standard                     │
│  Day 30:  → Move to S3 Standard-IA (cheaper)                │
│  Day 90:  → Move to S3 Glacier Flexible (archive)           │
│  Day 365: → Delete object                                    │
│                                                               │
│  Can also apply to:                                          │
│  - Current versions only                                     │
│  - Previous versions only (with versioning)                  │
│  - Objects with specific prefix (logs/*)                     │
│  - Objects with specific tags                                │
└──────────────────────────────────────────────────────────────┘
```

## BUCKET POLICIES & ACCESS CONTROL

```
┌─── Access Control Methods ───────────────────────────────────┐
│                                                               │
│  1. Bucket Policy (resource-based, JSON):                    │
│     - Applied to the bucket                                  │
│     - Can grant cross-account access                         │
│     - Can enforce encryption, IP restrictions                │
│                                                               │
│  2. IAM Policy (identity-based):                             │
│     - Applied to user/group/role                             │
│     - "This user can access S3"                              │
│                                                               │
│  3. ACLs (legacy, avoid):                                    │
│     - Object-level or bucket-level                           │
│     - Less flexible than policies                            │
│     - AWS recommends disabling ACLs                          │
│                                                               │
│  4. Block Public Access (account/bucket level):              │
│     - Safety net to prevent accidental public access         │
│     - ALWAYS enable unless you need public access            │
│     - Can be set at account level (applies to all buckets)   │
└───────────────────────────────────────────────────────────────┘

Example Bucket Policy — Allow public read (static website):
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": "*",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::my-website-bucket/*"
    }
  ]
}

Example — Force encryption on upload:
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Deny",
      "Principal": "*",
      "Action": "s3:PutObject",
      "Resource": "arn:aws:s3:::my-bucket/*",
      "Condition": {
        "StringNotEquals": {
          "s3:x-amz-server-side-encryption": "aws:kms"
        }
      }
    }
  ]
}
```

## REPLICATION

```
┌─── Cross-Region Replication (CRR) ──────────────────────────┐
│                                                               │
│  Source Bucket          Destination Bucket                    │
│  (ca-central-1)        (us-east-1)                          │
│  ┌──────────┐          ┌──────────┐                         │
│  │  photo   │ ────────►│  photo   │  (async replication)    │
│  │  .jpg    │          │  .jpg    │                         │
│  └──────────┘          └──────────┘                         │
│                                                               │
│  Requirements:                                               │
│  - Versioning must be enabled on BOTH buckets               │
│  - IAM role with replication permissions                     │
│                                                               │
│  Use for: disaster recovery, compliance, lower latency      │
└───────────────────────────────────────────────────────────────┘

┌─── Same-Region Replication (SRR) ────────────────────────────┐
│  Same as CRR but within same region                          │
│  Use for: log aggregation, live replication between accounts │
└───────────────────────────────────────────────────────────────┘
```

## STATIC WEBSITE HOSTING

```
S3 can serve a static website (HTML, CSS, JS)

URL format: http://bucket-name.s3-website-region.amazonaws.com

Setup:
1. Enable static website hosting on bucket
2. Set index document (index.html)
3. Set error document (error.html)
4. Add bucket policy for public read
5. Disable "Block Public Access"

For production: put CloudFront (CDN) in front
S3 → CloudFront → users (HTTPS, caching, custom domain)
```

## S3 PERFORMANCE

```
Baseline: 3,500 PUT/POST/DELETE + 5,500 GET per second per prefix

Multipart Upload:
- Required for files > 5GB
- Recommended for files > 100MB
- Uploads parts in parallel → faster

S3 Transfer Acceleration:
- Uses CloudFront edge locations
- Upload to nearest edge → AWS backbone → S3 bucket
- Good for long-distance uploads

S3 Select / Glacier Select:
- Query data inside S3 using SQL
- Only retrieve the data you need (not entire object)
- Up to 400% faster, 80% cheaper
```
