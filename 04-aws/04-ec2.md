## INSTANCE LIFECYCLE

```
┌──────────┐  launch   ┌─────────┐  stop    ┌─────────┐
│ Pending  │ ────────► │ Running │ ───────► │ Stopped │
└──────────┘           └─────────┘           └─────────┘
                            │                     │
                            │ terminate           │ start
                            ▼                     │
                       ┌──────────┐               │
                       │Terminated│ ◄─────────────┘
                       └──────────┘  (terminate)

Billing:
- Running  → you pay
- Stopped  → NO compute charge (EBS storage still charged)
- Terminated → nothing (instance gone, EBS deleted if set)
```

## INSTANCE TYPES — Deep Dive

```
Format: [family][generation].[size]
Example: t3.large, m6i.xlarge, c5.2xlarge

┌──────────┬──────────────┬──────────────────────────────────────┐
│ Family   │ Optimized For│ Use Cases                            │
├──────────┼──────────────┼──────────────────────────────────────┤
│ t3/t3a   │ Burstable    │ Dev/test, small web apps, microsvcs  │
│ m5/m6i   │ General      │ App servers, backend, medium traffic  │
│ c5/c6i   │ Compute      │ Batch processing, ML inference, HPC  │
│ r5/r6i   │ Memory       │ In-memory caches, real-time analytics│
│ i3/i4i   │ Storage      │ Data warehousing, distributed FS     │
│ g4/p4    │ GPU          │ ML training, video encoding, gaming  │
│ d2/d3    │ Dense Storage│ MapReduce, HDFS, log processing      │
└──────────┴──────────────┴──────────────────────────────────────┘

Sizes (each step = 2x resources):
nano → micro → small → medium → large → xlarge → 2xlarge → 4xlarge...

t3.micro:  2 vCPU,  1 GB RAM  (~$0.01/hr)
t3.large:  2 vCPU,  8 GB RAM  (~$0.08/hr)
m5.xlarge: 4 vCPU, 16 GB RAM  (~$0.19/hr)

Burstable (t3):
- Earns CPU credits when idle
- Spends credits when busy
- Good for variable workloads
- Unlimited mode: pay extra if credits run out
```

## PRICING MODELS

```
┌─── 1. On-Demand ─────────────────────────────────────────────┐
│  Pay per second (Linux) or per hour (Windows)                │
│  No commitment, no upfront cost                              │
│  Most expensive per hour                                     │
│  Use for: short-term, unpredictable, testing                 │
└───────────────────────────────────────────────────────────────┘

┌─── 2. Reserved Instances (RI) ───────────────────────────────┐
│  1 or 3 year commitment                                      │
│  Up to 72% discount vs On-Demand                             │
│                                                               │
│  Payment options:                                            │
│  - All Upfront:     biggest discount                         │
│  - Partial Upfront: medium discount                          │
│  - No Upfront:      smallest discount (still cheaper than OD)│
│                                                               │
│  Types:                                                      │
│  - Standard RI: locked to instance type (biggest discount)   │
│  - Convertible RI: can change instance type (less discount)  │
│                                                               │
│  Use for: steady-state workloads (databases, core app)       │
└───────────────────────────────────────────────────────────────┘

┌─── 3. Spot Instances ────────────────────────────────────────┐
│  Up to 90% discount                                          │
│  AWS can reclaim with 2-minute warning                       │
│  Use for: batch jobs, data analysis, CI/CD builds            │
│  NEVER for: databases, stateful apps                         │
│                                                               │
│  Spot Fleet: mix of Spot + On-Demand for availability        │
└───────────────────────────────────────────────────────────────┘

┌─── 4. Savings Plans ────────────────────────────────────────┐
│  Commit to $/hour for 1 or 3 years                          │
│  More flexible than RI (any instance family, region, OS)    │
│  Up to 72% discount                                         │
│  Types:                                                      │
│  - Compute Savings Plan: any instance, Lambda, Fargate      │
│  - EC2 Instance Savings Plan: specific instance family       │
└──────────────────────────────────────────────────────────────┘

┌─── 5. Dedicated ────────────────────────────────────────────┐
│  Dedicated Host: physical server for you only               │
│  Dedicated Instance: hardware not shared with others        │
│  Use for: compliance, licensing (per-socket, per-core)      │
│  Most expensive option                                       │
└──────────────────────────────────────────────────────────────┘
```

## STORAGE OPTIONS

```
┌─── EBS (Elastic Block Store) ────────────────────────────────┐
│  Network-attached disk (like a USB drive for EC2)            │
│  Persists after instance stop/terminate (if configured)      │
│  One EBS → one EC2 (except io1/io2 multi-attach)            │
│  Locked to one AZ (use snapshots to move across AZs)        │
│                                                               │
│  Volume Types:                                               │
│  ┌──────────┬──────────┬──────────┬──────────────────────┐  │
│  │ Type     │ IOPS     │ Use Case │ Cost                 │  │
│  ├──────────┼──────────┼──────────┼──────────────────────┤  │
│  │ gp3      │ 3000-16K │ General  │ $0.08/GB/mo (default)│  │
│  │ gp2      │ up to 16K│ General  │ $0.10/GB/mo (legacy) │  │
│  │ io2      │ up to 64K│ Database │ $0.125/GB/mo         │  │
│  │ st1      │ 500      │ Big data │ $0.045/GB/mo (HDD)   │  │
│  │ sc1      │ 250      │ Archive  │ $0.015/GB/mo (HDD)   │  │
│  └──────────┴──────────┴──────────┴──────────────────────┘  │
│                                                               │
│  gp3 is the default and best for most workloads             │
│  io2 for databases needing high IOPS                        │
│                                                               │
│  Snapshots:                                                  │
│  - Point-in-time backup of EBS volume                       │
│  - Stored in S3 (you don't see it)                          │
│  - Can copy across regions (disaster recovery)              │
│  - Can create new volume from snapshot in any AZ            │
└───────────────────────────────────────────────────────────────┘

┌─── Instance Store ───────────────────────────────────────────┐
│  Physically attached disk (not network)                      │
│  Very high IOPS (millions)                                   │
│  DATA LOST when instance stops or terminates                 │
│  Use for: temp data, cache, buffers                          │
│  NOT for anything you need to keep                           │
└───────────────────────────────────────────────────────────────┘

┌─── EFS (Elastic File System) ────────────────────────────────┐
│  Network file system (NFS) shared across multiple EC2        │
│  Auto-scales (no provisioning)                               │
│  Multi-AZ by default                                         │
│  Use for: shared content, CMS, home directories              │
│  Linux only                                                  │
└───────────────────────────────────────────────────────────────┘
```

### Placement Groups
```
┌──────────────┬──────────────────────────────────────────────┐
│ Cluster      │ All instances in same rack/AZ                │
│              │ Low latency, high throughput                  │
│              │ Risk: rack failure = all down                 │
├──────────────┼──────────────────────────────────────────────┤
│ Spread       │ Each instance on different hardware           │
│              │ Max 7 per AZ                                  │
│              │ For: critical instances that must be isolated │
├──────────────┼──────────────────────────────────────────────┤
│ Partition    │ Instances spread across partitions (racks)    │
│              │ Up to 7 partitions per AZ                     │
│              │ For: HDFS, HBase, Cassandra                   │
└──────────────┴──────────────────────────────────────────────┘
```

### Elastic IP (EIP)
```
┌─── Elastic IP ───────────────────────────────────────────────┐
│                                                               │
│  Problem: EC2 public IP changes every time you stop/start    │
│  Solution: Elastic IP = static public IP you own             │
│                                                               │
│  ┌──────────┐                                                │
│  │   EC2    │ ◄── Elastic IP: 54.23.1.100 (never changes)   │
│  │ Instance │                                                │
│  └──────────┘                                                │
│                                                               │
│  Rules:                                                      │
│  - 5 EIPs per region per account (can request more)          │
│  - FREE when attached to a running instance                  │
│  - CHARGED when NOT attached or instance is stopped          │
│    ($0.005/hour — AWS wants you to release unused EIPs)      │
│  - Can remap to another instance instantly (failover)        │
│                                                               │
│  Best practice:                                              │
│  - Avoid EIPs when possible                                  │
│  - Use ALB + DNS instead (more scalable)                     │
│  - EIP is for: bastion hosts, NAT instances, legacy apps     │
│    that need a fixed IP                                      │
│                                                               │
│  Commands:                                                   │
│  aws ec2 allocate-address                                    │
│  aws ec2 associate-address --instance-id i-xxx --allocation-id eipalloc-xxx │
│  aws ec2 release-address --allocation-id eipalloc-xxx        │
└───────────────────────────────────────────────────────────────┘
```

### ENI (Elastic Network Interface)
```
ENI = virtual network card attached to EC2

┌──────────┐
│   EC2    │──── ENI (eth0): primary, auto-created
│ Instance │──── ENI (eth1): secondary (optional)
└──────────┘

Each ENI has:
- Private IP (always)
- Public IP or Elastic IP (optional)
- Security groups
- MAC address

Use cases for multiple ENIs:
- Management network + application network
- Dual-homed instances (two subnets)
- Move ENI between instances (keep same private IP)
```

### AMI (Amazon Machine Image)
```
AMI = template to launch EC2 instances

┌──────────────┐   create AMI   ┌──────────────┐   launch
│ EC2 Instance │ ──────────────►│     AMI      │ ──────────► New EC2
│ (configured) │                │ (snapshot)   │   (identical copy)
└──────────────┘                └──────────────┘

Types:
- AWS provided: Amazon Linux, Ubuntu, Windows
- Marketplace: pre-configured (WordPress, Jenkins)
- Custom: your own (golden image with your app pre-installed)

AMIs are region-specific. Copy to other regions if needed.
```

## INTERVIEW QUESTIONS

**Q: Difference between stopping and terminating?**
```
Stop:      instance halted, EBS preserved, no compute charge
Terminate: instance deleted, EBS deleted (unless DeleteOnTermination=false)
```

**Q: How to make EC2 highly available?**
```
1. Use Auto Scaling Group across multiple AZs
2. Put ALB in front
3. Use launch template (not manual launch)
4. Health checks auto-replace unhealthy instances
```

**Q: EC2 instance can't connect to internet. Troubleshoot.**
```
1. Is it in a public subnet? (route table → IGW)
2. Does it have a public IP or Elastic IP?
3. Security Group allows outbound?
4. NACL allows outbound + ephemeral ports inbound?
5. Is the IGW attached to the VPC?
```
